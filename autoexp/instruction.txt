`autoexp` is a shared workspace for black-box agentic experimentation through a central execution plane.

## Agent Contract

When you create or edit an experiment, use this workflow:

1. Edit source files only inside `script/` or `autoexp.json`.
2. Put project-local environment values in `app.env`, never directly in scripts.
3. Execute the experiment only with `autoexp run`.
4. Read the `run_id:` line from command output.
5. Use that `run_id` for follow-up actions such as inspecting the run or running `autoexp run <run_id>`.

Do not run files inside `script/` directly. `autoexp run` copies the exact script/config state into a self-contained `runs/<run_id>/` directory, passes `app.env` into the selected runner, writes output/log artifacts there, and records the run history.
If the same capsule already produced identical stable output, Autoexp returns the existing `run_id` instead of recording a duplicate run.

Each completed or failed run is recorded in `index.sqlite`, retained under `runs/<run_id>/`, and available in the UI. Source and configuration state are snapshotted automatically when the run starts.

## Project Structure

- `script/` is the only versioned executable source directory.
- `script/stage.json` defines the runner command entry point.
- `script/params.json` contains user-editable non-secret parameters.
- `script/params.schema.json` describes those parameters for browser/API controls.
- `report.txt` contains project-specific report generation instructions. Edit it when the domain, audience, or report structure changes.
- `app.env` contains local environment variables and secrets. Autoexp creates it during project init, passes it to the selected runner, ignores it in `.gitignore`, and does not store it in the private Autoexp git history.
- `runs/` contains local run snapshots and generated artifacts.
- `.autoexp/` contains Autoexp's private git repository and is separate from any user-managed project-level git repository.

## Versioned Source

Autoexp computes a deterministic recursive content hash for `script/`, excluding files matched by the project `.gitignore`.

A run capsule is the combination of:

- `script_hash`
- `script_env_hash`
- `runtime_context_hash`

Changing files under `script/` changes the script hash. Changing `app.env` can change runtime behavior, but `app.env` itself is intentionally local and unversioned.

## Execution Trigger

Use `autoexp run` as the central execution trigger. Agents should call this command instead of running `script/` files directly.

Each run executes in `runs/<run_id>/` and receives a JSON context file at `${CTX}`. The run directory contains:

- `ctx.json`
- `run.json`
- `script/`
- `output/`
- `logs/`
- `report/`

Reports are not generated by `autoexp run`. Autoexp writes report context automatically at `runs/<run_id>/report/report_bundle.json` whenever a run id is recorded or refreshed.

When asked to generate a report, use the active report instructions from `report.txt` or the configured project report instruction file.
The bundle contains project-relative pointers to associated output and log files, not their contents. Read the referenced files needed for the report, then write generated report artifacts under `runs/<run_id>/report/`.
The report output may contain markdown, images, data files, or other generated assets.
Do not read secret values from `app.env`; use only the environment variable names included in the bundle.

## Runtime Context

Read paths from `${CTX}` instead of hardcoding runtime paths. Important keys:

- `script_dir`
- `script_params_path`
- `app_env_path`
- `output_dir`
- `logs_dir`

Generated output and logs must be written only to the paths provided by `${CTX}`. Do not write generated artifacts into `script/`.

## Editing Rules

- Put executable workflow logic, helpers, templates, and non-secret params in `script/`.
- Put local credentials, tokens, endpoints, and environment-specific values in `app.env`.
- Put sandbox, runtime, title, and metadata changes in `autoexp.json`.
- Do not edit generated files under `runs/` unless the user explicitly asks to inspect or repair artifacts.

From the terminal, `autoexp run <run_id>` refreshes that run's output/log/report-bundle artifacts in its existing `runs/<run_id>/` directory.
