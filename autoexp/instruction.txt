# Autoexp Project Contract

## Agent contract

1. Keep human- and agent-authored experiment files under `experiment/`.
2. Read `.autoexp/instructions.md` and use Autoexp MCP operations when available.
   - Use Standard mode for qualitative, comparative, multi-variant, or software experiments.
   - Use Autoresearch only when one stable numeric metric and a frozen evaluator can automatically decide keep versus revert.
3. Put local environment values in `.env`, never directly in source.
4. Run experiments through Autoexp so source, parameters, outputs, logs, and reports remain tied to an immutable run.
5. Never hand-edit completed run output or log artifacts.
6. Before the first recorded run, give generic source such as `main.py` or `candidate.py` a concise domain-specific filename when the project purpose makes one obvious. Do not leave the generic filename once the task is understood. Keep `.autoexp/stage.json` and the editable-source declaration in `.autoexp/project.json` aligned with the rename.
7. When the user's prompt defines the desired report audience, questions, comparisons, or sections, read the existing report guidance and append those project-specific requirements through the report-guidance UI or MCP capability before running. Do not discard the existing guidance.
8. Give every MCP-triggered run a short title that states the variant or hypothesis being tested. Prefer `SQLite warm-cache pass` over `Run 1` or an opaque identifier.
9. When the requested experimentation has distinct variants or phases that need separate reports and insights, use separate descriptively named scripts for those parts instead of one large parameter-switched script. Execute and report each part independently; share only genuinely common helpers.
10. Mark only decision-changing results, surprising failures, and new best results as milestones. At project completion, use the project summary evidence to write one central `.autoexp/project-report.md` for the end-to-end result.

## Workspace boundary

- `experiment/` contains the visible experiment source and domain files.
- `runs/` contains inspectable immutable run evidence and generated reports.
- `.env`, `.gitignore`, `AGENTS.md`, and `CLAUDE.md` remain user-editable root files.
- `.autoexp/` contains Autoexp-managed project settings, stage configuration, parameters, report guidance, state, and snapshot storage. Use the dedicated UI or MCP capability for these files rather than a generic script editor.
- `.autoexp/stage.json` defines the execution command.
- `.autoexp/params.json` and `.autoexp/params.schema.json` back the Parameters interface.
- `.autoexp/report-instructions.md` backs the Report Guidance interface.
- `.autoexp/project-report.md` is the agent-generated, project-level synthesis shown in both modes.

Only files declared by `.autoexp/project.json` as editable source belong in the script editing interface. Other files under `experiment/`, including datasets, are not implicitly editable there.

## Execution and evidence

Do not run experiment files directly when producing recorded evidence. `autoexp run` snapshots the exact experiment and managed configuration, copies that state into `runs/<run_id>/`, passes `.env` to the selected runner, and records the lifecycle in `.autoexp/state.sqlite`.

The experiment receives a JSON context path through `${CTX}`. Read `script_params_path`, `output_dir`, and `logs_dir` from that context. Write generated output only to `output_dir` and logs only to `logs_dir`.

Every run has an immutable source identity. Rerunning a prior run creates a child run from its pinned source. Restoring changes the workspace source; it does not rewrite historical evidence.

## Reports

Use the active report guidance exposed by Autoexp. A report bundle is written to `runs/<run_id>/report/report_bundle.json`; generated reports belong under that run's `report/` directory. Do not read secret values from `.env`; the bundle exposes names only.

Per-run reports explain individual executions. The central project report synthesizes the full experiment, including milestones and cross-run conclusions; generate it from `project_summary`, not from memory.

## Autoresearch

In Autoresearch mode, `experiment/program.md` contains human research direction, `experiment/candidate.py` is agent-owned, and `experiment/evaluate.py` is frozen within a research contract. Run research preflight first, change one hypothesis at a time, and finish every begun attempt through Autoexp. Kept and reverted candidates remain inspectable evidence. A changed evaluator starts a new contract boundary rather than rewriting old attempts.
